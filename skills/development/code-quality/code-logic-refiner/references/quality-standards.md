# 程式碼品質標準

本文件定義了 `code-logic-refiner` 技能在評估程式碼品質時應遵循的具體標準，旨在確保程式碼不僅功能正確，且具備高可維護性、低技術債和優良的性能。

## 1. 功能正確性與魯棒性

*   **完全符合需求**：程式碼必須完全實現所有功能需求，且行為與預期一致。
*   **邊界條件處理**：所有已知的邊界條件（例如：空輸入、最大/最小數值、異常字元等）都應得到妥善處理，避免程式崩潰或產生錯誤結果。
*   **錯誤處理機制**：應具備健壯的錯誤處理機制，能夠捕獲並妥善處理運行時錯誤、外部服務異常等，並提供有意義的錯誤訊息。
*   **安全性**：程式碼應避免常見的安全漏洞，例如 SQL 注入、跨站腳本 (XSS)、不安全的資料儲存等。

## 2. 可讀性與可維護性

*   **清晰的命名**：變數、函數、類別等命名應具備描述性，清晰表達其用途和含義。
*   **一致的程式碼風格**：遵循專案或語言的程式碼風格指南（例如：PEP 8 for Python, Google Java Style Guide），保持風格的一致性。
*   **適當的註解**：複雜的邏輯、非顯而易見的設計決策或潛在的陷阱應有清晰的註解說明。避免過度註解顯而易見的程式碼。
*   **模組化與解耦**：程式碼應劃分為獨立、高內聚、低耦合的模組或函數，便於理解、測試和重用。
*   **單一職責原則 (SRP)**：每個類別或函數應只負責一項功能，避免「萬能」類別或函數。

## 3. 簡潔性與高效性

*   **避免冗餘**：消除重複的程式碼 (DRY - Don't Repeat Yourself)，透過抽象化或函數重用來減少冗餘。
*   **簡潔的表達**：使用簡潔、直接的語法來表達邏輯，避免不必要的複雜性或迂迴的實現方式。
*   **性能考量**：在滿足功能和可讀性的前提下，應考慮程式碼的運行效率，避免明顯的性能瓶頸（例如：N+1 查詢、不必要的迴圈等）。
*   **資源利用**：合理利用計算資源（CPU、記憶體、網路頻寬），避免資源洩漏或過度消耗。

## 4. 技術債規避

*   **可測試性**：程式碼應易於編寫單元測試和整合測試，確保修改後不會引入新的問題。
*   **可擴展性**：設計應考慮未來的擴展性，避免硬編碼或難以修改的結構。
*   **依賴管理**：清晰定義和管理外部依賴，避免版本衝突或不必要的依賴。
*   **文件化**：除了程式碼註解，重要的設計決策、API 介面、部署流程等應有相應的文件記錄。

## 5. 跨 Agent 兼容性

*   **通用性指令**：SKILL.md 中的指令應盡量使用通用語言，避免過度依賴 Manus 特有的工具或環境。
*   **結構化輸出**：審查報告和建議應採用結構化的格式（例如 Markdown、JSON），便於不同 Agent 解析和理解。
*   **明確的輸入/輸出規範**：明確定義技能的輸入（例如：程式碼、需求文件）和輸出（例如：審查報告、修改建議）格式，以提高互操作性。
